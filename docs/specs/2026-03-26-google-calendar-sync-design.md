# Google Calendar Sync ŌĆö Design Spec

**Date:** 2026-03-26
**Status:** Approved

---

## Summary

Push planned workouts to a dedicated "Springa" Google Calendar alongside every Intervals.icu write. Strimma reads these events from the Android device calendar via CalendarProvider for pre-activity BG guidance and carb recommendations.

## Motivation

Strimma PR #77 adds pre-activity guidance ŌĆö reading upcoming workouts from the device calendar to show readiness assessments and carb guidance before a run. Intervals.icu does not sync to Google Calendar, so workouts generated by Springa are invisible to Strimma. This feature closes the loop:

```
Springa generates workout
  -> Google Calendar (new)
    -> Android device sync
      -> Strimma CalendarReader
        -> Pre-activity BG guidance + carb notification
```

---

## 1. Auth: Extend Existing Google OAuth

Springa already uses NextAuth with Google OAuth (`lib/auth.ts`). Extend it to request calendar access.

### Changes to Google provider config

- Add scope: `https://www.googleapis.com/auth/calendar` (full scope ŌĆö required to create the "Springa" calendar, not just events)
- Add `access_type: "offline"` to get a refresh token
- Dynamic `prompt: "consent"`: only set when the user has no valid `google_refresh_token` in the DB. This forces re-consent to obtain a refresh token. Once stored, subsequent sign-ins skip the consent screen.

### Token storage

- New column on `user_settings`: `google_refresh_token TEXT` (AES-256-GCM encrypted, same pattern as `intervals_api_key` and `mylife_password` in `lib/credentials.ts`)
- Populated in the NextAuth `signIn` callback via `{ user, account }` ŌĆö store `account.refresh_token` when present (Google only sends it on consent, not on regular sign-ins)
- Refresh tokens are long-lived (valid until revoked)

### Token exchange

- `getGoogleAccessToken(refreshToken)` exchanges refresh token for a short-lived access token via `https://oauth2.googleapis.com/token`
- Called before each Google Calendar operation ŌĆö no caching needed, operations are infrequent
- If token exchange fails (revoked, expired), log the error, clear the stored refresh token from the DB, and skip calendar sync silently. The cleared token triggers `prompt: "consent"` on the next sign-in, obtaining a fresh refresh token automatically.

## 2. Calendar Management

### Auto-create on first sync

On first sync, create a calendar named "Springa" via Google Calendar API:
```
POST https://www.googleapis.com/calendar/v3/calendars
{ "summary": "Springa", "timeZone": "<user timezone from user_settings>" }
```

Store the returned calendar ID in `user_settings.google_calendar_id TEXT`.

### Subsequent syncs

Reuse the stored calendar ID. If the API returns 404 (user deleted the calendar), recreate it and update the stored ID.

### No UI needed

The calendar is auto-created and auto-managed. The user sees it appear in Google Calendar and can toggle visibility, change color, etc. On Strimma's side, the user picks "Springa" from the calendar picker in Exercise Settings.

## 3. Sync Triggers

Every Intervals.icu write path gets a parallel Google Calendar write:

| User action | Intervals.icu call | Google Calendar call |
|-------------|-------------------|---------------------|
| Upload full plan | `uploadToIntervals()` ŌĆö bulk delete future + bulk create | `clearFutureGoogleEvents()` + `syncEventsToGoogle()` |
| Adapt sync | `updateEvent()` per adapted event | `findGoogleEvent()` + `updateGoogleEvent()` per event |
| Manual edit/move | `updateEvent()` | `findGoogleEvent()` + `updateGoogleEvent()` |
| Manual delete | `deleteEvent()` | `findGoogleEvent()` + `deleteGoogleEvent()` |
| Drag-drop move | `updateEvent()` | `findGoogleEvent()` + `updateGoogleEvent()` |

### Caller responsibility

The Google Calendar calls live alongside the Intervals.icu calls at the call site (PlannerScreen, EventModal, CalendarView, useDragDrop). This keeps `intervalsApi.ts` focused on Intervals.icu.

Each call site resolves the Google Calendar context **once** at the top of the operation, then passes `{ accessToken, calendarId }` to individual calls. This avoids redundant DB queries when updating multiple events (e.g., adapt sync updates N events but queries the DB only once).

## 4. Event Mapping

### Matching strategy

Springa owns the entire "Springa" calendar ŌĆö no other app writes to it. Workout names include the week number (e.g., "W01 Bonus Easy", "W05 Long eco16"), making name+date unique within a plan.

- **Bulk upload:** Clear all future events on the Springa calendar, then create new ones. No individual matching needed.
- **Manual edit/move:** Find the Google Calendar event by searching for `summary` (name) + `start.dateTime` (date) on the Springa calendar. Update the matching event. For moves, lookup uses the **old** name + **old** date, then patches the new date.
- **Manual delete:** Same lookup by name + date, then delete.
- **Adapt sync:** Each adapted event has `original.name` and `original.date` from the `CalendarEvent` ŌĆö use these for the lookup.

If the lookup finds no match (e.g., event was manually deleted from Google Calendar), skip silently.

### Event content

| Google Calendar field | Source |
|----------------------|--------|
| `summary` | `WorkoutEvent.name` (e.g. "Sun Long eco16") |
| `start.dateTime` | `WorkoutEvent.start_date_local` (with timezone) |
| `start.timeZone` | User's timezone from `user_settings.timezone` |
| `end.dateTime` | `start_date_local` + estimated duration (with timezone) |
| `end.timeZone` | User's timezone from `user_settings.timezone` |
| `description` | Formatted text (see below) |

### Description format

```
HR target: Z2 (135-153 bpm)
Fuel: 45 g/h

Warmup 10m 60%-70% LTHR (102-119 bpm)
Main 75m 70%-80% LTHR (119-136 bpm)
Cooldown 2km 60%-70% LTHR (102-119 bpm)
```

- First line: primary HR zone target for the workout category
- Second line: fuel rate in g/h (from `getCurrentFuelRate()`)
- Blank line, then full workout step breakdown (same as Intervals.icu description)

### Duration estimation

`estimateWorkoutDuration()` from `lib/workoutMath.ts` already computes total time from workout description steps. Used to set the event end time so the calendar block has the right visual length.

## 5. Architecture

### New file: `lib/googleCalendar.ts`

```
getGoogleAccessToken(refreshToken: string): Promise<string>
  ŌĆö POST to oauth2.googleapis.com/token, returns access_token

ensureSpringaCalendar(accessToken: string, storedCalendarId: string | null, timezone: string): Promise<string>
  ŌĆö GET calendar by stored ID, create if missing (with timezone), return calendar ID
  ŌĆö Does NOT write to the DB. Returns the calendar ID; the caller
    (getGoogleCalendarContext) writes it to user_settings if it changed.

syncEventsToGoogle(accessToken: string, calendarId: string, events: WorkoutEvent[], timezone: string): Promise<void>
  ŌĆö Create Google Calendar events sequentially from WorkoutEvents
  ŌĆö Sequential creation avoids Google Calendar API rate limits (~600 req/min).
    A typical plan has ~100 events; sequential takes a few seconds. Batch API
    can be added later if this becomes a bottleneck.

clearFutureGoogleEvents(accessToken: string, calendarId: string): Promise<void>
  ŌĆö List events from now onward on the Springa calendar, delete all
  ŌĆö Assumes <2500 future events (a plan has ~150 max). No pagination needed.

findGoogleEvent(accessToken: string, calendarId: string, name: string, date: string): Promise<string | null>
  ŌĆö Search by summary + date on the Springa calendar, return Google Calendar event ID or null

updateGoogleEvent(accessToken: string, calendarId: string, googleEventId: string, updates: EventUpdates): Promise<void>
  ŌĆö PATCH a Google Calendar event by its ID

deleteGoogleEvent(accessToken: string, calendarId: string, googleEventId: string): Promise<void>
  ŌĆö DELETE a Google Calendar event by its ID

formatEventDescription(event: WorkoutEvent, hrZones?: number[], lthr?: number): string
  ŌĆö Build the description text with HR target, fuel rate, and steps
```

### Helper: token + calendar resolution

A convenience function that resolves refresh token ŌåÆ access token ŌåÆ calendar ID in one call. Called **once per user action** (not per event):

```
getGoogleCalendarContext(userEmail: string): Promise<{ accessToken: string; calendarId: string } | null>
```

Steps:
1. Read `google_refresh_token` and `google_calendar_id` from `user_settings`
2. If no refresh token ŌåÆ return `null` (caller skips sync)
3. Decrypt refresh token, exchange for access token via `getGoogleAccessToken()`
4. If token exchange fails (401) ŌåÆ clear `google_refresh_token` from DB, return `null`
5. Call `ensureSpringaCalendar(accessToken, storedCalendarId, timezone)` to verify/create calendar
6. If calendar ID changed (new creation) ŌåÆ write new ID to `user_settings.google_calendar_id`
7. Return `{ accessToken, calendarId }`

### Adapt sync: data availability

The adapt sync path (`buildSyncPayload`) currently returns `{ eventId, description, fuelRate }`. To support Google Calendar updates, the call site needs access to the original event's `name` and `date` for the `findGoogleEvent` lookup. The `AdaptedEvent` already carries `original: CalendarEvent` with `.name` and `.date` ŌĆö the PlannerScreen sync handler uses both the `SyncPayloadItem` (for Intervals.icu) and the `AdaptedEvent` (for Google Calendar lookup) side by side. No changes to `buildSyncPayload` needed.

## 6. Schema Changes

`user_settings` table ŌĆö two new columns:

| Column | Type | Default | Purpose |
|--------|------|---------|---------|
| `google_refresh_token` | TEXT | NULL | Google OAuth refresh token (AES-256-GCM encrypted via `lib/credentials.ts`) |
| `google_calendar_id` | TEXT | NULL | ID of the auto-created "Springa" calendar (plaintext ŌĆö not sensitive) |

---

## 7. Error Handling

Google Calendar sync is best-effort. A failure must never block the Intervals.icu write.

- All Google Calendar calls wrapped in try/catch
- Errors logged to console
- UI shows a non-blocking toast: "Calendar sync failed ŌĆö workouts saved to Intervals.icu"
- Token revocation (401/403): clear `google_refresh_token` from DB ŌåÆ next sign-in triggers `prompt: "consent"` ŌåÆ fresh token obtained automatically
- Calendar deleted (404 from events API): `ensureSpringaCalendar` recreates, updates stored ID

## 8. Data Flow

```
User generates/adapts/edits plan
  |
  +-> uploadToIntervals(apiKey, events)        -> Intervals.icu
  |                                               -> Garmin Connect
  |                                                 -> Forerunner 970
  |
  +-> syncEventsToGoogle(accessToken, ...)     -> Google Calendar
                                                  -> Android device sync
                                                    -> Strimma CalendarReader
                                                      -> Pre-activity BG guidance
```

---

## 9. Not In Scope

- Reading from Google Calendar (Springa is write-only)
- Two-way sync with Intervals.icu (Intervals.icu remains source of truth)
- Settings toggle to enable/disable Google Calendar sync (always on if token exists; toggle can be added later)
- Calendar color/name customization (user can change these in Google Calendar directly)
- Recurring events (each workout is a standalone event)
- Event reminders/notifications (Strimma handles notifications via AlarmManager)
- Batch API for bulk event creation (sequential is sufficient for ~100 events)
